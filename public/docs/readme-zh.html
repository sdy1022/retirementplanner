1. 工具定位：它解决什么问题

退休规划真正困难的部分，不是算一个余额，而是同时处理“税、账户类型、领取时点、市场顺序、通胀、寿命和家庭状态变化”。本工具的核心价值，是把这些变量放进同一个年度模拟框架。

年度决策

估算每年应转换多少 Roth、从哪里支付生活费与税款、何时可能触发 RMD 或 IRMAA。

方案比较

比较“不转换、固定转换、填满税档、平滑收入、自动优化、SBLOC 税款融资”等不同策略。

风险压力测试

使用历史股债路径、历史 CPI、Guardrail 与 SSA 寿命表进行 Monte Carlo 模拟。

适合回答的问题包括：

  • 我应该在退休前或 RMD 前做 Roth conversion 吗？
  • 每年转换固定金额，还是填满某个税率档更合理？
  • 推迟退休、提高储蓄、改变股债比例，哪个更有效？
  • 如果经历高通胀、熊市或寿命超预期，计划还能维持吗？
  • Guardrail 提高成功率的代价，是不是降低了实际消费？
  • 配偶去世后，Single 税档、遗属 Social Security 与配偶 RMD 年龄会怎样影响计划？
重要：它是决策支持与情景分析工具，不是报税软件，也不是对未来资产的保证。最适合用于判断“方案方向和相对差异”，而不是把 30 年后的余额当作精确预测。

3. 账户管理

账户页面用于建立当前资产快照。模型不是只接收一个“总资产”，而是区分不同税务属性，以便正确计算税、RMD、取款顺序和遗产税后价值。

支持的账户类型

Traditional 401(k)Traditional IRARoth 401(k)Roth IRATaxable Brokerage

每个账户可记录

  • 账户名称：例如“Fidelity 401(k)”“Vanguard Roth IRA”。
  • 所有者：Primary、Spouse 或 Joint。
  • 账户类型与余额。
  • Brokerage cost basis：用于区分本金与未实现资本利得。
  • Snapshot date：标记余额日期。

同类型多个账户会在计算时正确聚合，例如多个 Traditional IRA、401(k) 或 Brokerage 账户。Brokerage 的余额和成本基础分别汇总，避免把全部卖出额误当成应税收益。

5. 确定性年度计算引擎

引擎以“每一年”为单位推进，从当前年龄模拟到最后幸存者的计划终点。每年大致执行以下顺序：

  1. 计算当年工资、其他收入、Social Security 与 RMD。
  2. 根据选定策略决定 Roth conversion 金额。
  3. 计算联邦税、州税、IRMAA 和可能的提前取款罚金。
  4. 按照现金流 waterfall 支付生活费、税与附加费用。
  5. 更新 Traditional、Roth、Brokerage 和成本基础。
  6. 应用投资收益、股息税拖累、供款及下一年度假设。
  7. 记录短缺、资产余额、边际税率及各资金来源。

年度输出字段

每年结果包括 Traditional / Roth / Brokerage 余额、Brokerage basis、RMD、conversion、taxable income、联邦税、州税、IRMAA、shortfall、边际税率、计划消费、实际消费和各类资金来源。

所有核心计算位于纯 TypeScript 函数中，独立于 Angular UI，因此可以单元测试、Worker 调用、Golden QA 回归，也便于审查计算顺序。

6. 税务、IRMAA 与 RMD

联邦税

  • 按累进税档计算，不是对全部收入使用同一边际税率。
  • 支持标准扣除和税档随政策假设指数化。
  • Conversion 只计入其引起的增量税负。
  • 州税目前按用户输入的平坦税率估算。

IRMAA

模型依据 MAGI 与两年滞后逻辑估算 Medicare Part B / D 附加保费。对于模拟开始时缺少历史年份的情况，可使用 preSimulationMagi 作为初始数据。

RMD

  • 使用 IRS Uniform Lifetime Table divisor。
  • SECURE 2.0 起始年龄根据出生年决定为 73 或 75。
  • Primary 去世当年仍按逝者规则处理；下一年开始，若配偶存活并完成 spousal rollover，则按配偶年龄与出生年计算 RMD。
  • 这项遗属切换逻辑避免继续错误地沿用 Primary 年龄。

7. Roth Conversion 策略

模式行为适合用途
None不进行 Roth conversion。作为基线。
Fixed Amount每年转换固定金额，可设置在 RMD 年龄或指定年龄停止。执行简单、可预算。
Fill to Bracket利用当年剩余税档空间，转换到目标税率档上沿。控制边际税率。
Smooth to Bracket在多个年份之间平滑转换，以避免未来收入或 RMD 尖峰。长期税率平滑。
Fill to Income将收入填至指定目标，可设置保留资产底线和停止年龄。控制 MAGI 或目标收入。
Smooth Income Target搜索更平滑的年度应税收入目标。降低税率、RMD 与 IRMAA 波动。
Auto Optimize遍历候选税档/策略并按税后评分选择。寻找候选最优方案。

工作期转换

可以允许退休前转换。模型先放入工资，再利用剩余税档空间做转换，并只计算 conversion 带来的增量税。这样可以评估“高工资时期是否仍值得转换”。

税后评分

策略选择不是简单比较税前总资产，而是考虑 Traditional 余额未来税负、Brokerage 未实现收益税率、Roth 的免税属性以及可能的 SBLOC 债务，从而得到更有意义的税后结果。

8. 现金流、供款和取款顺序

退休前积累

模型支持税前供款、Roth 供款、Brokerage 储蓄和 employer match。税后现金约束会限制不可能实现的 Roth / Brokerage 供款，避免“工资不够却仍按计划全部存入”的错误。

退休后支出 waterfall

生活费可由 Social Security、RMD、Traditional、Brokerage 和 Roth 依次提供。税款也有独立的资金来源记录，包括 Brokerage、conversion withholding、Traditional gross-up、Roth 或 SBLOC。

Spending Order

  • Traditional-first：先利用低税档从 IRA 取款。
  • Brokerage-first：先花 Brokerage，把税档空间留给 Roth conversion。
  • 自动选择：若未指定，模型比较两种路径，保留税后评分更高者。

Brokerage 细节

  • 按成本基础比例计算卖出中的本金和收益。
  • 股息收益率作为总收益的一部分，而不是额外叠加。
  • 股息每年产生税拖累，若再投资则提高 cost basis。
  • 终点 Brokerage gains tax 可设为 0 模拟继承 step-up，或设置为预计清算税率。

9. Social Security 与遗属阶段

  • Claim age 支持 62–70 岁。
  • 用户输入的是在选定 claim age 下的月度 benefit；引擎不会再次重复应用领取年龄系数。
  • Fixed inflation 模式使用用户配置的 ssColaRate。
  • Historical inflation 模式使用上一年度 sampled CPI 作为下一年度 COLA，并忽略固定 ssColaRate。
  • 配偶可配置独立 claim age 和 benefit。

第一位配偶去世后

下一年起模型切换到 Single filing，采用 Single 税档、扣除额、IRMAA 和一个 Medicare enrollee；Social Security 遗属收入保留夫妻两人中较高的一项潜在 benefit。账户余额转给幸存者。

这会显式模拟常见的“widow’s tax penalty”：家庭收入可能下降，但 Single 税档和 IRMAA 门槛也更低。

10. Buy-Borrow-Die 与 SBLOC 税款融资

在配置的年龄窗口内，Roth conversion 的增量税可不卖出 Brokerage，而是通过以证券为抵押的信用额度（SBLOC）借款支付。

  • 可设置开始年龄、结束年龄、借款利率和最大 LTV。
  • 贷款利息随余额复利增长。
  • 当贷款超过 collateral 支持能力时，模型执行 margin-call cure，优先使用 Brokerage，必要时使用 Roth 偿还。
  • 资产图可显示 gross assets；税后遗产分析应减去 SBLOC loan balance。
  • 策略选择器会比较现金缴税与 SBLOC 缴税的结果，不会假定借款一定更好。
SBLOC 具有利率、抵押品下跌、强制偿还和产品条款风险。本模型只用于压力测试概念，不代表任何贷款建议。

11. Monte Carlo 风险分析

Monte Carlo 页不是简单从正态分布生成独立收益，而是重放历史股债数据并通过 stationary block bootstrap 保留一部分连续年份关系。

核心设置与输出

输入

  • Trial 数量与 Seed
  • 股票配置比例
  • 固定或历史通胀
  • Guardrail 开关
  • 固定寿命或 SSA 随机寿命

输出

  • 计划成功概率
  • 期末资产 P10 / P50 / P90
  • 按年龄的资产 percentile fan chart
  • 短缺和消费实现率
  • Guardrail 触发统计

股债联合抽样

股票和 10 年期美国国债从同一历史年份抽取，并按用户配置的股票比例组合；模型每年再平衡。这样 60/40 组合不会把互不对应年份的股债收益随意拼接。

Web Worker

长时间模拟在 Web Worker 中运行，避免冻结浏览器 UI。Worker 会分批发送进度消息，而不是每个 trial 都发送。无 Worker 环境仍保留直接引擎 fallback，并用同一回调格式报告进度。

12. Fixed / Historical 双模式通胀

模式生活支出Social Security COLA用途
Fixed（默认）使用原有固定 3% 假设使用用户输入的 ssColaRate保持旧模型与 Golden 结果完全一致
Historical使用与股债同一历史年份的 CPI使用上一年度 sampled CPI模拟高通胀、低通胀和滞胀路径

联合年份一致性

Historical 模式下，某个 trial 年份的 stock return、Treasury return 和 CPI 来自同一个历史年份、同一个抽样区块。例如抽中 1974 年时，三者都取 1974 年的数据。

明确的模型边界

  • 税档和 IRMAA 门槛仍使用独立政策指数化，不直接等于 sampled CPI。
  • 策略搜索仍先在确定性计划上求解；随机通胀只用于 Monte Carlo trial 回放。
  • Guardrail 仍与固定通胀确定性基线比较，因此高通胀路径更容易触发削减，这是预期行为。

13. SSA 随机寿命

除了固定 life expectancy，工具还可以使用内嵌的 SSA 2023 Period Life Table 为每个 trial 抽样死亡年龄。

  • Primary 与 spouse 的寿命使用独立随机流。
  • Single 场景每个 trial 抽一个死亡年龄。
  • Married 场景分别抽样两人，第一人去世后切换遗属规则，模拟到最后幸存者。
  • 市场、Primary mortality、Spouse mortality 使用分离的确定性 seed namespace，确保复现性。

报告指标

  • Mortality-weighted success probability
  • Primary / spouse median death age
  • 最后幸存者年龄 P10–P90
  • 活到 90、95、100 岁的概率
  • Worker 与 direct engine parity
限制：SSA Period 表反映人口平均死亡率，不包含健康状况、吸烟、家族史、收入、医疗条件或个体风险。配偶死亡目前假设相互独立。

14. Adaptive Spending Guardrail

Guardrail 允许在不利市场路径中动态调整消费，以降低资金枯竭风险。它不是“免费提高成功率”，因为成功率上升可能来自减少支出。

正确的解读方式

  • 同时看成功率与 mean consumption realization。
  • 比较触发频率、削减幅度和恢复情况。
  • 不要只选择成功率最高的方案。

同 Seed 策略对比页专门把 Consumption 与 Success 并排展示，避免把“少花很多钱”误解成纯粹的策略改进。

15. 同 Seed 策略对比（Same-Seed Strategy Comparison）

它的目的，是让多个方案经历完全相同的随机世界，从而把差异尽可能归因于策略本身，而不是随机运气。

为什么必须同 Seed

如果方案 A 恰好抽到牛市，而方案 B 抽到熊市，即使 A 成功率更高，也不能说明 A 的策略更好。同 Seed 会让两者的第 1、2、3……个 trial 使用对齐的股票、债券、CPI 和随机流。

当前支持比较的变量

  • 退休年龄
  • 股票配置比例
  • 使用已保存的 Roth conversion 策略，或完全不转换
  • 是否启用 Guardrail
  • Seed 和 trial 数量

结果表

  • 成功概率及相对基准变化（百分点）
  • Median ending assets 及相对基准变化
  • 10th percentile ending assets
  • 消费实现率
  • 自动标识成功率最高的行
页面只复制 Scenario 后再改变指定变量，不会修改用户保存的原始 Scenario 或 Accounts。

16. Roth Conversion Tranche Planner

Tranche Planner 将一个年度 conversion 计划拆成多个执行检查点，帮助把“模型建议”转成更现实的操作节奏。

  • January：根据年初工资、税档和预期收入设置初始计划。
  • October / Fall True-Up：结合年内实际收入、奖金和市场变化修正。
  • December：在税年结束前完成最后校准。
  • Market Drawdown Check：市场明显回撤时，评估是否以较低资产价格提前转换。

它同时显示 bracket gauge、Traditional / Roth / Brokerage 只读余额、年度 conversion 目标和优化提示。

20. 推荐使用流程

  1. 录入账户。分别输入 Traditional、Roth 和 Brokerage，并填写 Brokerage cost basis。
  2. 建立基准 Scenario。填写当前年龄、出生年、退休年龄、工资、供款、开支、Social Security 和税率。
  3. 先跑确定性 Dashboard。确认现金流、RMD、conversion 和税务结果符合直觉。
  4. 建立保守/基准/乐观假设。改变收益率、退休年龄和支出，而不是只相信一个版本。
  5. 运行 Monte Carlo。先用 Fixed inflation 验证旧基线，再切 Historical inflation 查看滞胀风险。
  6. 打开 SSA Stochastic Longevity。观察死亡年龄分布对成功率的影响。
  7. 启用 Guardrail 后检查 Consumption。确认成功率提高是否伴随可接受的消费削减。
  8. 使用 Same-Seed Compare。比较推迟退休、改变配置、关闭 conversion 或开启 Guardrail。
  9. 查看 Year-by-Year Action Plan 和 Tranche Planner。将长期模型转换为本年度可执行动作。
  10. 每年重跑。更新余额、税表、工资、开支和 Social Security 估计，而不是一次计算后多年不变。

21. 如何正确解读结果

可信度分层

结果类型适合的解释不适合的解释
单年度税务机制用于检查税档空间、RMD、conversion sizing 和 IRMAA cliffs。不能代替正式报税或专业税务意见。
策略相对排名当差距很大且在多组假设下稳定时，通常具有决策价值。差距很小时不要把排名当作确定事实。
长期期末资产作为不同情景下的范围与风险指标。不能把几十年后的某个具体金额当成预测。

建议重点关注

  • 策略在 5%、6.5%、8% 等不同收益假设下是否方向一致。
  • 同 Seed 下方案差异是否稳定。
  • P10 结果是否仍可接受，而不只是 P50。
  • 成功率提升是否依赖显著削减消费。
  • 税后资产、累计税、RMD 和 IRMAA 是否同步改善。

22. 已知边界与免责声明

  • 不是财务、投资、法律、贷款或税务建议。
  • 税法、IRMAA、Social Security 和遗产规则可能变化。
  • 州税目前采用平坦税率近似。
  • Social Security 应税比例等部分规则存在简化。
  • 未必涵盖 NIIT、所有 deduction、credit、地方税和特殊账户规则。
  • SSA Period Life Table 不是个体化医学寿命预测。
  • 历史数据不能保证未来重复，block bootstrap 也不能涵盖所有宏观状态。
  • SBLOC 的真实利率、margin 条款和券商规则可能与模型不同。
  • 模型的精度受输入质量影响；错误的余额、cost basis、收入或 benefit 会直接影响结论。
使用原则：把工具用于发现风险、比较方向、准备问题和制定年度检查清单；在执行大额 conversion、借款、退休或 Social Security 决策前，咨询合格专业人士。

24. 模型假设与局限

成功率是模型条件下的模拟样本比例，不是个人现实世界中精确到 1% 的概率。确定性终值也不是未来资产承诺。最有价值的用途是比较方向、发现风险和准备年度决策。

重点假设

  • 历史股票、债券与 CPI 联合抽样保留历史关联，但未来不一定重复历史。
  • 同 Seed Compare 共享随机路径，用于减少比较中的抽样噪音。
  • SSA period life table 是人口级数据，不是个人医疗寿命预测。
  • 税务、IRMAA、Social Security、RMD 和遗属规则是规划模型，不能替代正式专业意见。
  • Guardrail 提高成功率可能伴随消费削减，必须同时查看 consumption realization。

v1.0 刻意未完整建模

包括 LTC 随机冲击、cohort mortality、个体健康、完整 Roth 五年 tranche 规则、QCD、复杂 inherited IRA、所有税法 sunset、非合格股息/税 lot 和完整供款合规校验。
